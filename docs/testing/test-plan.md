# テスト方針・実績報告書（Test Plan & Report）

本ドキュメントは、貸出管理システムにおけるテストの目的、アプローチ、環境、および実施結果についてまとめたものです。

---

## 1. テストの目的
本プロジェクトにおけるテスト活動の目的は以下の通りです。

* **ドメインルールの保護**
  貸出・返却・期限超過など、システムの核となるビジネスルール（ドメイン知識）の堅牢性を担保するため、特にドメイン層の単体テストに注力します。
    * *目的:* 状態遷移や異常系の挙動を100%予測可能にすること。
* **デグレード（先祖返り）の防止**
  将来的な新機能追加やリファクタリング時に、既存機能が破壊されていないかを自動テストで即座に検知できるようにします。
* **仕様書としての機能**
  テストコード（JUnit 5 / AssertJ）を読むことで、システムの正しい振る舞い（仕様）がコードレベルで理解できる状態にします。
* **外部依存の切り離し（今後サービス層テストで導入予定）**
  Mockitoを活用してリポジトリ層（DBアクセス）などの依存関係をモック化し、サービス層独自のビジネスロジックのみを純粋かつ高速に検証できるようにします。

---

## 2. テスト対象機能
本システムにおけるテストの主要な対象範囲です。

* **貸出管理機能 (`Loan` ドメイン)**
    * 資産の貸出処理、二重貸出の防止ロジック
    * 予定返却日および期限超過（`OVERDUE`）の判定ロジック
* **返却管理機能 (`returnAsset` 制御)**
    * 返却処理、実際返帰日の記録
    * すでに `RETURNED` 状態の場合の「べき等性（スルー処理）」の検証
* **資産・会員管理機能 (`Asset`, `User` ドメイン)**
    * 各エンティティの生成、バリデーションルール

### 2.1 重点検証項目（品質のこだわり）
特にシステムの根幹となる以下のロジックについては、異常系や境界値を含め重点的に検証しています。
* **ドメイン駆動設計（DDD）におけるビジネスルール:**
  「期限超過した資産は新規貸出不可」など、Entity内に閉じ込めたロジックの網羅（Domain層 C1カバレッジ実績あり）。

### 2.2 今回対象外とした項目（今後の課題）

> [!WARNING]
> **同時実行時の排他制御（本バージョンではスコープ外）**
> 同一データへの同時アクセス（例：残り1つの資産への同時貸出申請など）に対するデータベースレベルの排他制御（楽観的ロックなど）は、本バージョンのスコープ外としています。今後の機能拡張にて対応予定です。

## 3. テスト戦略（ピラミッド型アプローチ）
本プロジェクトでは、テストの自動化効果（費用対効果）を高めるため、レイヤーごとの役割を分離して開発を進めています。

### ① 単体テスト (Unit Test) — 【最優先・網羅性重視】
* **Domain層テスト (実装済):** 外部依存を100%排除。ビジネスルールや状態遷移の全分岐（C1カバレッジ）を検証済。
* **Repository層テスト (今後着手予定):** `Testcontainers` を導入。ローカルの Docker 上に使い捨ての PostgreSQL コンテナを起動し、実際の SQL 構文や DB 制約の挙動を検証。
* **Application/Service層テスト (今後着手予定):** `Mockito` を用いてRepository層をモック化。業務シナリオの正常系および、ビジネスルール違反時の例外処理（準正常系）を検証。
* **Web/Controller層テスト (今後着手予定):** `MockMvc` を使用し、HTTPステータスコード、JSONマッピング、入力バリデーション（桁数・必須チェック等）を検証。

### ② 結合テスト (Integration Test) — 【連携重視・ケースは厳選】
* **ユースケース結合テスト (E2E / 今後着手予定):** コントローラーから物理DBまでモックを使わずに結合。主要なハッピーパス（正常系シナリオ）および、代表的な業務エラー（準正常系）のライフサイクルが正しく動作することを確認。

## 4. テスト環境・使用ツール
テスト実行および品質測定に使用した技術スタックです。

* **テストフレームワーク:** JUnit 5
* **検証（アサーション）:** AssertJ （可読性の高い流れるようなインターフェースのため採用）
* **モックライブラリ (今後着手予定):** Mockito （外部サービスやRepositoryのモック化）
* **カバレッジ測定:** JaCoCo (Java Code Coverage)
* **テスト用データベース:** Testcontainers (PostgreSQL)
* **CI/CD自動化 (今後着手予定):** GitHub Actions (Pull Request作成時に自動で `mvn test` を実行)
