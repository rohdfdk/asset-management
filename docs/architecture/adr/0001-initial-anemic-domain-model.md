# ADR 0001: 初期設計における貧血ドメインモデルと手続き型サービス層の採用

## 1. コンテキスト（背景と課題）
プロジェクト発足当初、迅速にプロトタイプ（動くもの）を作成し、動作検証を行う必要があった。
開発者自身のオブジェクト指向およびDDD（ドメイン駆動設計）への習得度が発展途上であったため、最も直感的かつ迅速に実装できるアーキテクチャを選択する必要があった。

## 2. 意思決定（下した決断）
ドメインオブジェクト（`Asset` 等）は、ゲッターとセッター（Getter/Setter）のみを持つ「データを保持するための単なる器」とし、すべてのビジネスロジック（バリデーション、状態遷移の判定、計算）はサービス層（`AssetService`）に手続き型として記述する。

つまり、**「貧血ドメインモデル（Anemic Domain Model）」**を採用する。

## 3. ステータス
拒絶 / 過去の記録（Superseded by ADR-0002）

## 4. 結果（メリットとデメリット）
### メリット
- 実装が極めて直感的（上から下に手順を書くだけ）であり、初期の機能実装スピードは非常に速かった。
- データベースのテーブル構造とドメインオブジェクトが1対1で対応するため、O/Rマッパーとの親和性が高かった。

### デメリット（後に発生した重大な課題）
- **ロジックの散乱：** 資産の状態遷移チェックが複数のサービスやコントローラーに重複して書かれるようになり、仕様変更の影響範囲が制御不能になった。
- **不正な状態の発生：** どこからでも `asset.setStatus(RETIRED)` のように状態を上書きできてしまうため、ビジネスルールを無視した不正なデータがDBに保存されるリスクが発生した。
- **テストの肥大化：** サービス層のテストを書くために、DBのMockや複雑なテストデータのセットアップ（前提条件の構築）が大量に必要となり、TDDのサイクル（検証スピード）が著しく低下した。

## 5. 次のアクション
本設計による技術負債が許容範囲を超えたため、ドメイン層をリッチ化し、カプセル化を強制する設計へとリファクタリングを行う（詳細は **ADR-0002** にて決定）。
