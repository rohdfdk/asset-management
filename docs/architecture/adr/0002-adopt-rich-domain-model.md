# ADR 0002: サービス層の手続き型設計を廃止し、リッチドメインモデルを採用する

## 1. コンテキスト（背景と課題）

初期実装では、ビジネスロジック（資産の状態遷移など）をサービス層に集約していた。

その結果、以下の課題が発生した：

- 状態遷移ルールがサービス層に分散し、変更時の影響範囲が広がる
- ドメインモデルが振る舞いを持たず、データ構造のみの存在になる（貧血ドメイン）
- 仕様変更のたびに複数の層に修正が発生する

## 2. 意思決定

サービス層でビジネスロジックを保持する設計を廃止し、
リッチドメインモデル（DDD）を採用する。

ビジネスルールおよび状態遷移の制御は、
`Asset` などのドメインエンティティに集約する。

## 3. ステータス
Accepted

## 4. 結果（メリットとトレードオフ）

### メリット

- ビジネスルールがドメイン層に集約され、変更影響範囲が限定される
- ドメインモデル単体でのテストが可能になり、単体テストの独立性が向上
- 状態遷移の一貫性が保証される

### トレードオフ

- 手続き型よりも設計初期の認知負荷が高い
- ドメイン設計の理解が前提となるため学習コストが増加する
