# ADR 0002: サービス層の手続き型設計を廃止し、リッチドメインモデルを採用する

## 1. コンテキスト（背景と課題）
初期の実装では、ビジネスロジック（資産の状態遷移など）をすべてサービス層に記述していた。
しかし、仕様変更（例：MAINTENANCEからRETIREDへの遷移追加）のたびにサービス層のif文が肥大化し、コードが陳腐化（スパゲティ化）する課題に直面した。また、ドメインがただの箱（貧血症）になり、カプセル化が機能していなかった。

## 2. 意思決定（下した決断）
サービス層での手続き型の実装を禁止し、DDD（ドメイン駆動設計）の思想に基づく「リッチドメインモデル」を採用する。ビジネスルールと状態遷移のバリデーションは、すべて `Asset` などのドメインエンティティおよびValue Object内に閉じ込める。

## 3. ステータス
承認（Accepted）

## 4. 結果（メリットとデメリット）
### メリット
- ビジネスルールが1箇所にカプセル化され、仕様変更（状態遷移の追加など）の影響範囲がドメイン層のみに閉じるようになった。
- ドメイン層を純粋に関数としてテストできるようになり、TDDによるC1網羅率100%の達成が容易になった。
### デメリット / トレードオフ
- 直感的な手続き型の思考（上から下に順番に書くコード）から脱却する必要があり、設計時の認知負荷（行ったり来たりする時間）が増加する。
